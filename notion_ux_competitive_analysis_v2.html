Mixed Checklist is task-driven ("Create your first task") — directive but motivating
Machine No in-product onboarding copy. Assumes admin-led setup.

Help docs (macro)
Human Conversational, use-case framed. "How to build a project tracker."
Mixed Task-framed help. Step-by-step but dense with feature names.
Machine Reference-style. Parameters, API fields, admin flags. Minimal narrative.
Mixed Community-written docs vary. Official docs lean technical.
Empty state copy (micro)
Human "A fresh page for your ideas" — invites creation without pressure
Mixed "No tasks yet — add one!" — directive and functional
Machine "No pages exist in this space" — system confirmation, not invitation
Human "Build something here" — builder metaphor, empowering

Mixed Friendly tone but vague cause — "Something went wrong" without recovery path
Machine Generic system errors. Automation failures often silent.
Machine Atlassian stack traces shown to end users. Verbose and technical.
Mixed Formula errors are precise for devs, opaque for non-technical users.

Page tree hierarchy vs block-based flexibility — cognitive load impact
Notion — block-based flexibility
Model
Everything is a block — text, images, databases, embeds. Pages contain blocks, blocks contain other blocks.
Navigation
Freeform sidebar tree. No enforced structure. Users organise as they see fit — unlimited nesting depth.
Cognitive load
Medium at startLow after 1 week Blank canvas is disorienting initially. Once mental model clicks, navigation is frictionless.
Flexibility cost
High creative freedom = high disorganisation risk. Teams without shared conventions end up with chaotic sidebar trees.
UX win
Win
Drag-and-drop block reordering reduces restructuring friction to near zero — lowest IA edit cost of the four.
UX issue
Issue
No enforced IA means shared workspaces degrade over time. No guardrails for team consistency.
ClickUp — enforced 5-level hierarchy
Model
Workspace → Space → Folder → List → Task. Every item has a fixed position in the hierarchy. Cannot be bypassed.
Navigation
Rigid tree. Views (list, board, Gantt) are overlays on the same hierarchy — not alternative structures.
Cognitive load
High at startMedium ongoing 5 levels must be understood before any work can begin — steep initial cognitive demand.
Flexibility cost
Predictable IA for large teams but bureaucratic for small ones. Simple projects require creating unnecessary hierarchy nodes.
UX win
Win
Multiple views (list, board, calendar, Gantt) on the same data set is genuinely powerful — same IA, different lenses.
UX issue
Issue
Hierarchy labels ("Spaces", "Folders", "Lists") don't match how teams actually think about their work — vocabulary mismatch increases cognitive load.
Confluence — page tree hierarchy
Model
Site → Space → Page → Child page. Content lives in a strict tree. No block model — pages are document units.
Navigation
Left sidebar page tree. Tree can grow to hundreds of nodes — no auto-archiving or visibility capping.
Cognitive load
High and sustained Tree nav becomes unmanageable at scale. Finding content requires either good memory or functional search — search is unreliable.
Flexibility cost
Lowest flexibility of the four. Page-centric model struggles with dynamic or relational data — workarounds required.
UX win
Win
Page hierarchy is instantly legible to new users — tree nav requires no onboarding, just exploration.
UX issue
Issue
No equivalent to Notion's inline database — data and docs are structurally separated, requiring context-switching between Confluence and Jira.
Coda — doc-as-app with table-first IA
Model
Workspace → Doc → Page → Section. Tables are first-class IA elements — can reference and filter across the entire doc.
Navigation
Left sidebar by doc, then page within. Flat and clean at doc level but no cross-doc navigation.
Cognitive load
Highest initialLow for power users Requires inversion of expectations — a "doc" behaves more like an app than a document.
Flexibility cost
Highest flexibility of the four for structured data. Tables with cross-references rival relational databases. Text editing is secondary.
UX win
Win
Cross-table references within a doc eliminate data duplication — strongest relational IA of the four products.
UX issue
Issue
No cross-doc navigation or linking at the IA level — teams with multiple docs must manually maintain context between them.
